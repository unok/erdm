# SDD要件レビュー規約

要件ドキュメントの品質レビューのルール。

## 原則

| 原則 | 基準 |
|------|------|
| 品質保証 | 完璧を求めず、許容可能なリスクで判断する |
| 重大問題集中 | 指摘は最大3件。設計・実装に大きく影響するものだけ |
| バランス | 問題点だけでなく強みも認める |
| 実行可能性 | すべての提案は修正可能であること |

## レビュー観点

### EARS形式への準拠（最重要）

- 各受け入れ条件がEARSパターン（イベント駆動/状態駆動/望ましくない振る舞い/任意機能/常時）に従っているか
- 1条件1振る舞いが守られているか
- 要件IDが数値のみ（N.M形式）か

### 要件の完全性

- 機能要件・非機能要件が網羅されているか
- 正常系・異常系・境界条件がカバーされているか
- 暗黙の前提が明示化されているか

### 要件の一貫性

- 要件間に矛盾がないか
- 用語の統一が取れているか
- 要件間の依存関係が明示されているか

### テスト可能性

- 各受け入れ条件が検証可能か
- 曖昧な表現（「適切に」「迅速に」「十分に」等）が排除されているか
- 具体的な基準・数値が定義されているか

## 判定

| 判定 | 条件 |
|------|------|
| GO | EARS形式に準拠、要件が網羅的、テスト可能、重大な矛盾なし |
| NO-GO | EARS形式の不備、重大な要件漏れ、テスト不可能な条件、要件間の矛盾 |

## トレーサビリティ

- 各指摘を該当する要件IDに紐づける
- 具体的な改善提案を提示する
